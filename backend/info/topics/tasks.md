## Tasks list {#list}

Tasks are grouped into **Individual**, **Group**, and **Cohort**:

- **Individual** — personal tasks you created or staff assigned to you; your own status per enrollment.
- **Group** — one shared status for the whole group.
- **Cohort** — same assignment for many students; each student has their own status.

**Private** tasks hide the title from teachers; they may still see metadata (status, due date) for students in their groups.

## Task detail {#detail}

Status, priority, due date, subtasks, comments, and updates (when enabled). Staff-assigned tasks may lock editing of title/description for students.

## Create task {#form-create}

### Title (required)

Name of the task in lists and on the detail page.

### Description

Optional longer text: steps, links, definition of done.

### Priority

- **Low / Normal / High** — affects sorting and badges only, not who can see the task.

### Due date

Optional calendar date; shown in tables. Missing a due date does not block completion.

### Status (edit or some student personal tasks)

**To do · Doing · Blocked · Done** — your progress on this task (or the group’s single status on shared group tasks).

### Assignment (staff only, new task)

Choose one mode:

**Individual students**

- Pick **Cohort** or **Group**, then select the target.
- Check students to enroll — each gets their **own status** on the same task definition.
- Use for homework-style work where everyone progresses separately.

**Whole group (shared)**

- Pick one group — **one status** for all members (e.g. “Prepare stand-up presentation”).
- Students in that group all see the same task row on the Group section of the list.

### Subtasks (mini-tasks on the form)

Lines you add before saving become **template subtasks** on the task detail page (when subtasks are allowed). Each subtask can have **priority** and an optional **due date**. Students track **To do · Doing · Blocked · Done** per subtask (their own progress on enrolled tasks). They may add personal subtask lines if the task allows participant subtasks. **Edit** and **Delete** on a subtask row follow the same rules as the parent task: staff manage template subtasks; students manage only subtasks they added.

**Tasks vs goals:** goals use simple milestone checkboxes; task subtasks behave like small work items with status and metadata.

### Sections (checkboxes)

Control what appears on the **task detail** page after you save:

**Allow progress updates**

- **On** — a timeline where the student (or each enrollment) can post short progress notes with timestamps.
- **Off** — no updates area; use status changes only.

**Allow comments**

- **On** — threaded discussion on the detail page (good for questions to the teacher).
- **Off** — no comment thread.

**Allow subtasks**

- **On** — template subtasks + optional student-added items with per-student status.
- **Off** — no subtasks UI on detail.

Typical combos: all on for coached assignments; all off for a simple private reminder.

### Visibility

**Students**

- **Only me** — private: teachers see status/dates but not title or description.
- **Share with teachers** — full task visible to supervising teachers.

**Staff**

- **Draft (staff only)** — students do not see the task until you change to Share with students.
- **Share with students** — appears in student task lists immediately.

## Edit task {#form-edit}

Same fields where your role allows. Students on staff-assigned tasks often cannot change title/description but can change status and use enabled sections (updates, comments, subtasks).

## Delete task {#form-delete}

Removes the task and enrollments. Group/cohort tasks affect many people.

## For students {#for-students}

Use **Individual** for personal work. **Group** tasks share one status — agree in the team before marking Done.

## Admin notes {#admin-only}

<!-- role: admin -->

Admins see public tasks and private metadata on the dashboard, not private bodies.
